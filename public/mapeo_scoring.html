Mapeo de Integración: Formulario Web ➔ Airtable

Este documento especifica la estructura de datos del Formulario de Viabilidad/Scoring de Hipoteca Aquí, definiendo cómo se sincronizan y procesan los datos a través de las Netlify Functions con la base de datos de Airtable.

1. Tabla: Contacts (Contactos)

Registra la información del primer titular y, en caso de existir y completarse, crea de forma automática un segundo registro de contacto para el Titular 2.

3. Consulta de Viabilidad y Scoring (Airtable ➔ Web)

Una vez insertado el registro en Airtable, la web consulta el estado realizando peticiones periódicas mediante Polling a la Netlify Function check-scoring. Se leen los siguientes campos calculados automáticamente en Airtable para pintar el resultado en el frontend:

Campo en Airtable Tipo Destino / Uso en la Web
Viabilidad Single Select / Text Determina si el flujo es viable de forma automática. Si incluye "viable" (y no "no viable"), se muestran los importes. Si no, muestra la ventana de Revisión Manual.
Mejor cuota Fija Currency Pinta el importe mensual de la mejor alternativa de cuota fija.
Mejor cuota Mixta Currency Pinta el importe de la mejor alternativa de cuota mixta.
Mejor cuota Variable Currency Pinta el importe de la mejor alternativa de cuota variable.
Nº viables Number Indica cuántas entidades financieras pre-aprueban la operación en el texto explicativo.
SemaforoEstabilidad Text (Emoji) Representa con un semáforo de color (🟢, 🟡, 🔴) la estabilidad laboral del cliente en la ventana de revisión manual.
SemaforoEsfuerzo Text (Emoji) Semáforo visual de la capacidad de endeudamiento del perfil.
Semafor20masgatos Text (Emoji) Semáforo visual que evalúa si los ahorros aportados alcanzan el 20% más los gastos.
⚠️ Regla Crítica de Sincronización

Los valores enviados desde el frontend en formato numérico (Edad, Ingresos, Ahorros, etc.) deben limpiarse y parsearse a tipos válidos (enteros o flotantes) antes de enviarse a Airtable para evitar errores de tipo en la API.
